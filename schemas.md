# Schemas And Payloads

This document defines shared payload conventions for all SDKs.

## Request Schema Source

Each endpoint in `HttpEndPointData_t` includes a JSON schema-like object. CORE uses that metadata in two places:

1. `GET /getEndpoints` returns it as the `params` field
2. `GET /openapi.json` exposes it as request body schema metadata

When CORE has a schema for an endpoint, it validates JSON request bodies against that schema.

If validation fails, the server returns `400` with a plain-text message beginning with `Schema validation failed:`.

## Request Content Types

Current implementation behavior is mixed:

- Some endpoints accept query parameters on `GET`
- Some `POST` endpoints expect JSON request bodies
- Some endpoints explicitly require `Content-Type: application/json`

SDK guidance:

- Send `application/json` for all JSON body requests
- Avoid form encoding unless a specific endpoint explicitly requires it
- Preserve query-string parameters for `GET` endpoints that advertise them

## Response Shapes

There is no single global response envelope enforced across the local CORE API.

Current patterns in code include:

- Bare success with handler-defined body or no body
- JSON objects such as `{ "success": true, "message": "..." }`
- JSON result objects with endpoint-specific fields
- Plain-text error responses generated by the API layer

SDK guidance:

- Do not assume every success response contains `success: true`
- Do not assume every failure response is JSON
- Parse by `Content-Type` first, then fall back to raw text when needed

## Common JSON Conventions

Although not globally enforced, current endpoint implementations commonly use:

- `success` as a boolean
- `message` as a human-readable string
- resource-specific fields such as `auth_code`, `token`, or `cube_socket_path`

SDKs should preserve raw payload access even when exposing typed wrappers.

## Schema Normalization Guidance

Each SDK should normalize the discovery schema into a common internal representation with:

- endpoint path
- HTTP method
- public/private flag
- request schema
- human description

This allows the language-specific SDK to generate:

- typed request models
- typed response helpers where possible
- validation errors with endpoint context
- docs or code generation later

## Minimal Cross-SDK Assumptions

These assumptions are safe across languages:

- Request schemas are JSON-oriented
- Path names are exact and case-sensitive
- Method choice is explicit, not inferred
- Response bodies may be JSON, text, binary, or empty

## Example Discovery Fields

`GET /getEndpoints` returns per-endpoint metadata using these current keys:

- `name`
- `params`
- `description`
- `public`
- `endpoint_type`

SDKs should preserve these raw values in discovery APIs even if they also expose a cleaner typed model.
